❌ 기존 방식의 문제

SerializeField 직접 바인딩

일반적으로 Unity에서 적(Enemy)이나 플레이어(Player) 같은
프리팹을 사용하려면 [SerializeField]로 Inspector에
직접 드래그해서 연결합니다.

🚨 새로운 적 종류를 추가할 때마다
→ 스크립트에 새 필드를 만들고
→ Inspector에서 일일이 다시 연결해야 하고
→ 실수로 빠뜨리면 런타임 에러 발생

적 10종, 투사체 5종, 이펙트 8종...
콘텐츠가 늘어날수록 관리 비용이 폭발적으로 증가!

게임의 규모가 커질수록 이 방식은
유지보수가 불가능한 수준이 됩니다.

❌ 다른 대안들도 불편

Resources 폴더 / Asset Bundle

Unity에는 기존에도 에셋을 불러오는 다른 방법이 있지만,
각각 단점이 있습니다.

📁 Resources 폴더
· 문자열로 경로를 써서 로드 → 오타 시 에러
· Resources 안의 모든 파일이 빌드에 포함
· 안 쓰는 파일도 포함되어 앱 용량 증가
· 공식적으로 Unity도 "사용을 지양하라"고 권고

📦 Asset Bundle
· 번들 생성·관리·버전 관리를 전부 직접 구현
· 의존성 관리가 매우 복잡
· 로드/언로드 코드를 직접 작성해야 함
· 설정과 코드량이 많아 소규모 팀에 부담

배경 — Addressable Assets 도입
✅ 해결책: Addressable Assets

Unity의 Addressable Assets은 쉽게 말해 "이름표(라벨)를 붙여놓고, 필요할 때 이름만 불러서 가져오는 시스템"입니다.

✨ 왜 도입하는가?

1. 콘텐츠 추가가 간편
　새로운 적 프리팹을 만들면 "enemy" 라벨만 붙이면 끝.
　코드 수정 없이 자동으로 게임에 포함됩니다.

2. 메모리 효율적
　필요한 순간에만 로드하고, 안 쓰면 자동 해제.
　게임이 무거워지는 것을 방지합니다.

3. 라벨 기반 일괄 로드
　"skill_config" 라벨이 붙은 에셋을 한 번에 전부 로드.
　개별 경로를 외울 필요가 없습니다.

📋 프로젝트 적용 예시

// 기존: 일일이 연결 [SerializeField] GameObject slime; [SerializeField] GameObject goblin; [SerializeField] GameObject dragon; // 적 추가될 때마다 필드 추가...
▼
// Addressable: 라벨만 지정하면 끝 Addressables.LoadAssetsAsync<GameObject>( "enemy", // ← 이 라벨만 붙이면 됨 (prefab) => { pool.Register(prefab); } ); // 새 적을 추가해도 코드 변경 불필요!

🏠 로비 시스템

게임 시작 전 캐릭터와 맵을 선택하는 준비 화면입니다.

  • 캐릭터 선택 — 좌우 버튼으로 캐릭터를 넘기며 선택. 3D 모델 미리보기 제공
  • 캐릭터 해금 — 조건을 달성하면 새로운 캐릭터가 해금됩니다
  • 맵 선택 — 여러 맵 중 도전할 맵을 선택. 맵마다 스테이지 수와 적 구성이 다릅니다
  • 장비 세팅 — 배럴(특수 공격)과 오브(공전 투사체) 등 장비를 변경할 수 있습니다

💡 캐릭터 정보는 Addressable Assets의 라벨 기반 로딩으로
새 캐릭터 추가 시 코드 수정 없이 자동 반영됩니다.

게임 시스템 ③ — 전투 시스템

⚔️ 자동 전투

  • 플레이어는 가장 가까운 적을 자동으로 탐지하여 투사체를 발사합니다
  • 플레이어는 이동만 조작하면 되므로 직관적인 조작감을 제공합니다
  • 공격 속도, 투사체 수 등은 스킬에 의해 강화됩니다

💀 피격 · 사망

  • 적에게 닿으면 데미지를 받고, 짧은 무적 시간이 부여됩니다
  • 체력이 0이 되면 게임오버 팝업이 표시됩니다
  • 재도전 또는 로비 복귀를 선택할 수 있습니다

🎯 투사체 시스템

  • 기본 화살 투사체를 발사하며, 스킬에 따라 다양하게 변형됩니다
  • 멀티샷 — 여러 발을 동시에 발사
  • 대각선샷 — 좌우 대각선 방향으로 추가 발사
  • 수평샷 — 좌우로 추가 투사체 발사

🔥 속성 · 상태이상 효과

  • 화염(Fire) — 일정 시간 동안 지속 데미지
  • 독(Poison) — 느린 지속 데미지
  • 빙결(Ice) — 이동 속도 감소
  • 번개(Lightning) · 마그마(Magma)

📈 레벨업

  • 적을 처치하면 경험치 보석이 드롭됩니다
  • 경험치가 가득 차면 레벨업하며, 게임이 일시정지됩니다
  • 3개의 스킬 중 1개를 선택하여 캐릭터를 강화합니다

🧩 스킬 종류 (예시 — Addressable 라벨 기반으로 자유롭게 추가 가능)

  • 공격속도 증가 — 투사체 발사 간격 감소
  • 이동속도 증가 — 캐릭터 이동 속도 증가
  • 멀티샷 — 기본 공격 화살 수 증가
  • 대각선샷 — 좌우 대각선 방향 추가 투사체
  • 수평샷 — 좌우 수평 방향 추가 투사체
  • 아이스 배럴 — 범위 공격 (투하형)
  • 아이스 오브 — 캐릭터 주위를 도는 빙결 투사체
  • 베놈 오브 — 독 속성 공전 투사체

💡 스킬은 ScriptableObject로 정의되어 새 스킬 추가 시 코드 수정 불필요

👾 적의 태그 시스템 (Bit Flag)

적은 비트 플래그 기반 태그를 조합하여 정의됩니다. 하나의 적이 여러 태그를 동시에 가질 수 있습니다.

🏷 유형 태그 (Identity)

태그 설명
Melee 근접형 적
Ranged 원거리형 적
Boss 보스 — 높은 체력, 특수 패턴

🦶 이동 모듈 (Move)

모듈 설명
FollowMove 플레이어를 향해 추적 이동
PatternMove 정해진 경로를 순회하며 이동
RandomMove 랜덤 방향으로 이동

💡 예: Boss + FollowMove + Shoot + Fire → 화염 원거리 보스

⚔️ 공격 모듈 (Attack)

모듈 설명
MeleeAttack 근접 타격 공격
FollowMeleeAttack 추적 후 근접 공격 (돌진형)
Shoot 투사체 발사 (원거리)
FlyingShoot 공중 투사체 발사

속성 효과
Fire 화염 — 지속 데미지
Ice 빙결 — 이동속도 감소
Poison 독 — 느린 지속 데미지

🤖 적 상태 머신 (FSM)

각 적은 Idle · Move · Attack · Hurt · Die
5가지 상태를 가지며, 조건에 따라 전이합니다.

📋 EnemyBrain (ScriptableObject)

상태 전이 흐름은 EnemyBrain이라는
ScriptableObject에 정의됩니다.

[Header("State Flow")] [SerializeField] EnemyState afterIdle; [SerializeField] EnemyState afterMove; [SerializeField] EnemyState afterAttack; [SerializeField] EnemyState afterHurt;

💡 각 적마다 서로 다른 EnemyBrain SO를
할당하여 상태 흐름을 다르게 설정할 수 있습니다.

예시:
· 근접 적: Idle → Move → Attack
· 원거리 적: Idle → Attack (이동 없이 즉시 공격)
· 보스: Idle → Move → Attack (특수 패턴)

🗺️ 맵 구성

  • 각 맵은 여러 개의 스테이지로 구성됩니다

📊 스탯 시스템

캐릭터 스탯은 3개 계층으로 나뉩니다:

  • 기본(Base) — 캐릭터 고유 능력치
  • 장비(Equipment) — 장비 착용 시 영구 보너스
  • 인게임 버프(Buff) — 스킬 선택 시 해당 판 한정 효과

📐 전체 구조

게임의 모든 핵심 기능은 매니저(Manager)라는 단위로 분리되어 있고,
매니저들끼리는 이벤트 버스(EventBus)를 통해 서로 소통합니다.
이를 통해 각 기능이 독립적으로 동작하며, 수정이 쉬운 구조를 만듭니다.

UI 계층 — 화면에 보이는 모든 팝업·버튼
매니저 계층 — 게임 흐름을 제어하는 16개 매니저
게임플레이 계층 — 플레이어·적의 실제 동작
데이터 계층 — JSON, Addressable로 데이터 관리
인프라 계층 — 오브젝트 풀, 이벤트 버스, 입력 처리

기술 역할
Addressables 에셋을 라벨로 관리, 콘텐츠 추가 시 코드 수정 불필요
오브젝트 풀링 적·투사체·이펙트를 재사용하여 성능 최적화

Async/Await 데이터 로딩을 비동기로 처리, 로딩 화면 구현
